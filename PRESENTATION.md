# Сириус.Аренда — план презентации

Ориентир по времени: 5-7 минут на защиту + вопросы. Ниже - план по пунктам,
каждый можно превратить в отдельный слайд или просто рассказывать по порядку.

---

## 1. Суть проекта (30 секунд)

«Сириус.Аренда» - бэкенд-сервис для бронирования переговорных комнат и
учебных пространств университета. Реализован как REST API на FastAPI с
понятным веб-интерфейсом поверх него.

Ключевая задача, которую решает сервис: не дать двум людям забронировать
одно и то же пространство на пересекающееся время - и сделать это заметно
через понятную ошибку, а не молча.

## 2. Функциональность (1 минута)

Обязательный минимум по заданию:
- CRUD для пространств (`/rooms`)
- Бронирование с проверкой пересечений (`/bookings`)
- Просмотр расписания комнаты на дату
- JSON-хранилище данных
- Понятные HTTP-статусы и сообщения об ошибках (400/404/409)

Сверх минимума (доп. баллы по заданию):
- JWT-аутентификация с разделением прав - отменить бронь может только её
  автор
- Поиск свободных пространств по интервалу и вместимости (`/rooms/available`)
- 6 юнит-тестов, покрывающих: создание, конфликт бронирования, права на
  отмену, 404, поиск свободных, список «моих броней»
- Docker + Docker Compose
- Полноценный веб-интерфейс - не просто список комнат и форма, а весь
  пользовательский сценарий: регистрация → поиск → бронь → отмена

## 3. Технологический стек и почему именно так (2 минуты)

| Технология | Почему выбрана |
|---|---|
| **FastAPI** | Явно указан как приоритетный вариант в задании. Даёт бесплатную автогенерацию Swagger/OpenAPI-документации (обязательный пункт задания), встроенную валидацию через Pydantic и асинхронность из коробки - не нужно вручную писать сериализацию или проверку типов. |
| **Pydantic-схемы** | Разделение моделей на `Create` / `Update` / публичное представление - типовой подход, который не даёт случайно отдать наружу лишние поля (например, хэш пароля) и валидирует данные до того, как они попадут в бизнес-логику. |
| **JSON-файлы вместо БД** | Задание явно разрешает такой вариант как «упрощённый». Для объёма данных учебного проекта полноценная БД избыточна; вместо неё - своя обёртка `JSONTable` с блокировками (`threading.Lock`) и атомарной записью через временный файл, чтобы не терять данные при параллельных запросах. Архитектура спроектирована так, что переход на SQLite/PostgreSQL — это замена одного модуля (`storage.py`), а не переписывание всего проекта. |
| **JWT (PyJWT) + PBKDF2** | JWT - стандартный подход для REST API (не хранит состояние сессии на сервере). Пароли хранятся не в открытом виде, а как PBKDF2-HMAC-SHA256 с солью — это защита без внешних бинарных зависимостей вроде `bcrypt`, что упрощает установку на любой системе. |
| **Vanilla JS для интерфейса** | Не выбирался тяжёлый фронтенд-фреймворк, потому что весь интерфейс отдаётся статикой прямо из FastAPI (`StaticFiles`) - ноль дополнительной инфраструктуры сборки, простой деплой одним контейнером. |

## 4. Ключевое техническое решение — проверка пересечений (1 минута)

Алгоритм в `app/services.py`: для двух интервалов времени `[start, end)`
пересечение есть, если одновременно верно:
```
start < existing_end  И  end > existing_start
```
Эта проверка выполняется при каждом создании брони (`POST /bookings`) и при
поиске свободных пространств (`GET /rooms/available`) — то есть используется
одна и та же функция в двух местах, что исключает рассинхронизацию логики.

## 5. Демонстрация через Swagger UI (2–3 минуты)

Порядок для живого показа (или для скриншотов, если демо офлайн):

1. Открыть `/docs` - показать автосгенерированную документацию, разделы
   `auth`, `rooms`, `bookings`.
2. **POST `/auth/register`** → зарегистрировать пользователя → код `201`.
3. **POST `/auth/login`** → получить токен → нажать «Authorize» вверху
   страницы, вставить токен.
4. **POST `/rooms`** → создать пространство → код `201`, показать тело
   ответа с `id`.
5. **GET `/rooms`** → показать список созданных пространств.
6. **POST `/bookings`** → создать бронь на конкретное время → код `201`.
7. **Повторить тот же запрос `POST /bookings`** с теми же временем и
   комнатой → получить **`409 Conflict`** с сообщением «Пространство уже
   забронировано на выбранное время» — это самый наглядный момент, стоит
   выделить его отдельно.
8. **GET `/rooms/available`** с этим же интервалом → показать, что
   забронированная комната не попадает в список.
9. **GET `/bookings/mine`** → список своих броней.
10. **DELETE `/bookings/{id}`** → отмена → код `204`.
11. Повторить `GET /rooms/available` → показать, что комната снова
    свободна.

### Что заскриншотить для отчёта/презентации

Сделай скриншоты (в Swagger UI, вкладка ответа сервера) для:
- [ ] `POST /auth/register` - успешный ответ `201`
- [ ] `POST /rooms` - успешный ответ `201` с телом
- [ ] `POST /bookings` - успешный ответ `201`
- [ ] Повторный `POST /bookings` — ответ **`409`** с сообщением о конфликте
      (это главный скриншот, показывающий ключевое требование задания)
- [ ] `GET /rooms/available` - список без забронированной комнаты
- [ ] `DELETE /bookings/{id}` - ответ `204`
- [ ] Общий вид `/docs` со списком всех эндпоинтов

Плюс 1–2 скриншота веб-интерфейса (`/`) - шаг с бронированием и шаг «Мои
брони» с кнопкой отмены, чтобы показать не только API, но и удобство
использования.

## 6. Тесты (30 секунд)

```
pytest -v
```
6 тестов: создание/список комнат, конфликт бронирования (409), права на
отмену чужой брони (403), 404 на несуществующую комнату, поиск свободных
пространств с исключением занятых, список «моих броней» только для своего
пользователя.

## 7. Возможные вопросы комиссии — и короткие ответы

- **«Почему не SQL-база?»** - Задание разрешает JSON как упрощённый
  вариант; хранилище спроектировано модульно, замена на SQLite потребует
  изменений только в `storage.py`.
- **«Что будет при одновременных запросах?»** - Каждая JSON-таблица
  защищена `threading.Lock`, запись атомарна (через временный файл и
  `replace`), это исключает повреждение данных при параллельных запросах.
- **«Почему JWT, а не сессии?»** — REST API без состояния на сервере —
  стандартный подход, упрощает горизонтальное масштабирование в будущем.
- **«Как проверяются права на отмену брони?»** - В каждой брони хранится
  `owner_id`; при отмене сверяется с `id` текущего авторизованного
  пользователя, при несовпадении — `403 Forbidden`.
