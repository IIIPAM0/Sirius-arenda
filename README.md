# Сириус.Аренда — сервис бронирования пространств

REST API на **FastAPI** для бронирования переговорных комнат и учебных
пространств. Хранилище данных — JSON-файлы (`app/data/*.json`).

Всё можно делать прямо в браузере через понятный веб-интерфейс — искать
свободные пространства, бронировать, отменять брони — без необходимости
вручную работать со Swagger или curl.

## Возможности

- CRUD для пространств (`/rooms`)
- **10 готовых пространств создаются автоматически** при первом запуске —
  регистрируйся и сразу выбирай, без необходимости создавать комнаты вручную
- Бронирование с проверкой пересечений по времени (`/bookings`)
- Просмотр расписания комнаты на дату (`/rooms/{id}/bookings?date=...`)
- **Расписание-таблица на день** (`/rooms/schedule?date=...`) — сетка
  «пространство × час» по всем комнатам сразу, с рабочими часами 08:00–22:00
- Поиск свободных пространств по интервалу и вместимости (`/rooms/available`)
- Список «моих броней» (`/bookings/mine`) — для удобного отображения в интерфейсе
- JWT-аутентификация (`/auth/register`, `/auth/login`) — управлять своими
  бронированиями может только их владелец
- Юнит-тесты (`tests/test_api.py`)
- Docker / Docker Compose
- Понятный веб-интерфейс (`/`) с пошаговым сценарием: войти → найти
  пространство → забронировать → посмотреть свои брони → посмотреть расписание
- Автодокументация Swagger UI: `/docs`, ReDoc: `/redoc` (не обязательна для
  использования, но пригодится, если нужно посмотреть API «в сыром виде»)

## Зависимости

Указаны в `requirements.txt`, устанавливаются одной командой `pip install -r requirements.txt`:

| Пакет | Назначение |
|---|---|
| `fastapi` | веб-фреймворк, реализует REST API |
| `uvicorn[standard]` | ASGI-сервер, на котором работает приложение |
| `pydantic` | валидация входных/выходных данных и схемы |
| `PyJWT` | генерация и проверка JWT-токенов авторизации |
| `python-multipart` | разбор form-data (нужно для `POST /auth/login`) |
| `pytest` | запуск юнит-тестов |
| `httpx` | HTTP-клиент, используется `TestClient` в тестах |

Требуется **Python 3.11–3.12** (на 3.14 некоторые зависимости пока не
собираются из готовых пакетов — см. раздел «Быстрый запуск»).

## Быстрый запуск (Windows)

Дважды кликните на файл **`start.bat`** в папке проекта — он сам создаст
виртуальное окружение (если его ещё нет), установит зависимости и запустит
сервер. Как только в открывшемся окне появится строка
`Uvicorn running on http://127.0.0.1:8000`, откройте в браузере:

```
http://127.0.0.1:8000/
```

Не закрывайте это окно, пока пользуетесь сайтом — в нём работает сервер.

## Запуск вручную (PowerShell)

```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

uvicorn app.main:app --reload
```

Приложение будет доступно на `http://127.0.0.1:8000`.

## Запуск в Docker

```bash
docker compose up --build
```

## Запуск тестов

```bash
pytest -v
```

## Как пользоваться веб-интерфейсом

Страница `http://127.0.0.1:8000/` устроена как последовательность из 4 шагов:

1. **Войдите или зарегистрируйтесь** — переключатель «Войти / Зарегистрироваться»
   наверху страницы.
2. **Найдите свободное пространство** — укажите дату/время (обычным выбором
   даты, без ручного набора текста) и нажмите «Показать свободные». Там же
   можно добавить новое пространство.
3. **Забронируйте** — выберите пространство из выпадающего списка (или
   нажмите «Забронировать это» прямо на карточке комнаты) и подтвердите.
   Если время уже занято, сайт покажет понятное сообщение и не даст создать
   пересекающуюся бронь.
4. **Мои брони** — список ваших бронирований с кнопкой «Отменить» у каждой
   активной брони. Чужие брони не отображаются и не могут быть отменены.
5. **Расписание на день** — таблица «пространство × час» по всем комнатам
   сразу: зелёным — свободные часы, красным — занятые. Обновляется
   автоматически после каждого бронирования или отмены.

## Документация API (справочник эндпоинтов)

Живая интерактивная версия этой документации всегда доступна на `/docs`
(Swagger UI) и `/redoc` (ReDoc) при запущенном сервере — там же можно
отправлять запросы прямо из браузера.

Токен передаётся в заголовке: `Authorization: Bearer <access_token>`.

### Аутентификация

| Метод и путь | Тело / параметры | Ответ | Ошибки |
|---|---|---|---|
| `POST /auth/register` | JSON: `{"username": str, "password": str (мин. 6 символов)}` | `201` → `{"id": int, "username": str}` | `400` — имя уже занято или пароль слишком короткий |
| `POST /auth/login` | form-data: `username`, `password` | `200` → `{"access_token": str, "token_type": "bearer"}` | `401` — неверный логин/пароль |

### Пространства (`/rooms`)

| Метод и путь | Авторизация | Тело / параметры | Ответ | Ошибки |
|---|---|---|---|---|
| `POST /rooms` | требуется | JSON: `{"name": str, "capacity": int (>0), "equipment": [str]}` | `201` → объект `Room` | `401` |
| `GET /rooms` | нет | query: `min_capacity` (int, необяз.), `equipment` (список str, необяз.) | `200` → список `Room` | — |
| `GET /rooms/available` | нет | query: `start` (datetime ISO), `end` (datetime ISO), `capacity` (int, необяз.) | `200` → список `Room`, свободных на интервал | `400` — `end` раньше или равен `start` |
| `GET /rooms/{room_id}` | нет | путь: `room_id` (int) | `200` → объект `Room` | `404` — не найдено |
| `PUT /rooms/{room_id}` | требуется | JSON, любые поля из `{"name","capacity","equipment"}` | `200` → обновлённый `Room` | `401`, `404` |
| `DELETE /rooms/{room_id}` | требуется | путь: `room_id` | `204` — без содержимого | `401`, `404` |
| `GET /rooms/{room_id}/bookings` | нет | query: `date` (YYYY-MM-DD) | `200` → список `Booking` на эту дату | `404` — комната не найдена |
| `GET /rooms/schedule` | нет | query: `date` (YYYY-MM-DD, необяз. — по умолчанию сегодня) | `200` → объект `DaySchedule`: сетка «пространство × час» (08:00–22:00) со статусом `free`/`booked` для каждого часа каждой комнаты | — |

**Объект `Room`:**
```json
{ "id": 1, "name": "Переговорная 101", "capacity": 6, "equipment": ["Проектор", "Доска"] }
```

### Бронирования (`/bookings`)

| Метод и путь | Авторизация | Тело / параметры | Ответ | Ошибки |
|---|---|---|---|---|
| `POST /bookings` | требуется | JSON: `{"room_id": int, "start_time": datetime ISO, "end_time": datetime ISO, "username": str}` | `201` → объект `Booking` | `401`; `404` — комната не найдена; `409` — время пересекается с другой активной бронью |
| `GET /bookings/mine` | требуется | — | `200` → список `Booking` текущего пользователя | `401` |
| `DELETE /bookings/{booking_id}` | требуется | путь: `booking_id` | `204` — без содержимого | `401`; `403` — чужая бронь; `404` — не найдена; `400` — уже отменена |

**Объект `Booking`:**
```json
{
  "id": 1,
  "room_id": 3,
  "start_time": "2026-08-01T10:00:00",
  "end_time": "2026-08-01T11:00:00",
  "username": "Иван Иванов",
  "status": "active",
  "owner_id": 1
}
```
`status` принимает значения `"active"` или `"cancelled"`.

## Как пользоваться API напрямую (для тестирования/проверки)

1. Зарегистрироваться: `POST /auth/register` — `{"username": "...", "password": "..."}`
2. Войти: `POST /auth/login` (form-data: `username`, `password`) → получить `access_token`
3. Передавать токен в заголовке `Authorization: Bearer <token>` для
   операций, требующих авторизации

### Пример: создать комнату

```bash
curl -X POST http://127.0.0.1:8000/rooms \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"name": "Переговорная 101", "capacity": 6, "equipment": ["Проектор", "Доска"]}'
```

### Пример: забронировать

```bash
curl -X POST http://127.0.0.1:8000/bookings \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"room_id": 1, "start_time": "2026-08-01T10:00:00", "end_time": "2026-08-01T11:00:00", "username": "Иван Иванов"}'
```

При пересечении времени с существующей активной бронью сервер вернёт
`409 Conflict`.

## Схема данных

**Room** - `id, name, capacity, equipment[]`
**Booking** - `id, room_id, start_time, end_time, username, status (active|cancelled), owner_id`
**User** - `id, username, password_hash`

## HTTP-статусы

| Код | Когда |
|---|---|
| 200 | успешный запрос |
| 201 | объект создан |
| 204 | успешное удаление/отмена |
| 400 | ошибка валидации |
| 401 | не авторизован |
| 403 | нет прав (например, чужая бронь) |
| 404 | объект не найден |
| 409 | конфликт бронирования |

## Примечания по реализации

- Пароли хранятся как PBKDF2-HMAC-SHA256 (соль + 100 000 итераций) —
  без внешних бинарных зависимостей вроде bcrypt.
- JWT — HS256, срок жизни токена 24 часа. `SECRET_KEY` задаётся переменной
  окружения (см. `docker-compose.yml`); в проде обязательно задайте свой.
- JSON-хранилище защищено `threading.Lock` на каждый файл и делает
  атомарную запись через временный файл + `replace`, чтобы избежать
  повреждения данных при параллельных запросах.
- Токен авторизации в веб-интерфейсе хранится в `localStorage` браузера —
  вход сохраняется между перезагрузками страницы, пока не нажата кнопка
  «Выйти».
- Рабочие часы (08:00–22:00) заданы константами `WORK_START_HOUR` /
  `WORK_END_HOUR` в `app/storage.py` — единые для всех пространств.
- 10 стандартных пространств создаются один раз при первом запуске
  (`seed_default_rooms()`, вызывается в `main.py` на событии `startup`) —
  если пространства уже есть, сидинг ничего не делает повторно.
